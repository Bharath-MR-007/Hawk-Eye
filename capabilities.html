Monitoring Checks

All checks are configured in checks.yaml and are dynamically reloaded at runtime without restarting the application.

1. HTTP Health Check

Performs periodic HTTP GET requests to a list of targets and records their availability (up/down) and response time. Supports HTTP/1.1 and HTTPS with custom TLS certificates.

  • Configurable check interval (e.g., every 10s, 30s, 60s)
  • Custom timeout per target
  • Automatic retries with exponential backoff
  • Works with HTTP and HTTPS targets
YAMLchecks.yaml — Health check example 
health:
  interval: 30s
  timeout: 10s
  retry:
    count: 3
    delay: 2s
  targets:
    - https://api.example.com/health
    - https://auth.example.com/ping
    - http://internal-service:8080/status

Prometheus Metrics produced:

Metric Type Description
hawkeye_health_up Gauge 1 = target is up, 0 = target is down. Labelled by target

2. Latency Check

Measures and records the Round-Trip Time (RTT) between this Hawk-Eye instance and remote targets. Especially powerful in multi-region deployments where you run one Hawk-Eye node per data center and they probe each other.

  • RTT measured in milliseconds, exported as a Prometheus histogram
  • P50, P95, P99 percentile calculation built-in via Prometheus
  • Can target any HTTP/HTTPS endpoint, including other Hawk-Eye instances
YAMLchecks.yaml — Latency check example 
latency:
  interval: 60s
  timeout: 10s
  targets:
    - https://hawk-eye-us-east.internal
    - https://hawk-eye-eu-west.internal
    - https://api.thirdparty.com
Metric Type Description
hawkeye_latency_duration_seconds Gauge Latency with status (DEPRECATED)
hawkeye_latency_seconds Gauge Current latency to target
hawkeye_latency_count Counter Total number of latency checks performed
hawkeye_latency_duration Histogram Distribution of latency values over time

3. DNS Resolution Check

Monitors domain name resolution performance including lookup time, NXDOMAIN detection, and DNS server reliability. Critical for environments that depend on internal DNS for service discovery.

YAML
dns:
  interval: 60s
  timeout: 5s
  targets:
    - api.internal.company.com
    - database.internal
    - www.google.com
Metric Type Description
hawkeye_dns_status Gauge 1 = resolved, 0 = failed
hawkeye_dns_check_count Counter Total DNS checks performed
hawkeye_dns_duration_seconds Gauge Time taken to resolve the domain
hawkeye_dns_duration Histogram Distribution of DNS resolution times

4. DNS Advanced Check

An enhanced DNS check that supports configurable query types (A, AAAA, MX, CNAME, TXT, etc.) and multiple upstream resolvers. Especially useful for validating internal DNS infrastructure and comparing results across resolvers.

  • Specify multiple resolvers (e.g., 8.8.8.8, 1.1.1.1, internal resolvers)
  • Configurable query types per domain
  • Labels results by target, type, and resolver
YAMLchecks.yaml — DNS Advanced example 
dns_advanced:
  interval: 60s
  timeout: 5s
  resolvers:
    - 8.8.8.8:53
    - 1.1.1.1:53
  queries:
    - name: api.internal.company.com
      type: A
    - name: mail.company.com
      type: MX
Metric Type Description
hawkeye_dns_advanced_status Gauge 1 = resolved, 0 = failed. Labels: target, type, resolver
hawkeye_dns_advanced_duration_seconds Gauge Duration of the DNS query in seconds. Labels: target, type, resolver

5. Traceroute Check

Performs a layer 3 hop-by-hop path analysis to targets using ICMP or TCP. Identifies exactly which router/hop is the source of latency or packet loss. Requires the NET_RAW capability in Docker.

YAML
traceroute:
  interval: 120s
  timeout: 3s
  maxHops: 30
  targets:
    - addr: 8.8.8.8
      port: 53
    - addr: api.example.com
      port: 443
Metric Type Description
hawkeye_traceroute_minimum_hops Gauge Minimum number of hops to reach the target
hawkeye_traceroute_packet_loss_pct Gauge Percentage of hops that did not respond (0–100)
hawkeye_traceroute_check_duration_ms Gauge Total time taken to complete the traceroute check in milliseconds

6. TCP Meter Check

Measures TCP connection establishment time and reachability for any host:port target. Useful for validating firewall rules, service ports, and network-level connectivity without relying on HTTP.

  • Tests raw TCP connectivity (not HTTP)
  • Measures exact handshake duration in seconds
  • Works for any protocol port (80, 443, 5432, 6379, etc.)
YAMLchecks.yaml — TCP Meter example 
tcp_metrics:
  interval: 30s
  timeout: 5s
  targets:
    - database.internal:5432
    - redis.internal:6379
    - api.example.com:443
Metric Type Description
hawkeye_tcp_status Gauge 1 = TCP connection succeeded, 0 = failed. Label: target
hawkeye_tcp_connect_seconds Gauge Time to establish TCP connection in seconds. Label: target

7. SSL/TLS Validity Check

Connects to HTTPS/TLS endpoints and inspects the certificate's expiry date. Sends early warnings before certificates expire to prevent outages. Automatically tracks all configured host:port targets.

  • Reports days remaining until certificate expiry
  • Sets status to 0 (failure) if the connection or handshake fails
  • Works alongside the health check for full L7 + TLS visibility
YAMLchecks.yaml — SSL/TLS example
ssl_tls:
  interval: 3600s
  timeout: 10s
  targets:
    - api.example.com:443
    - auth.internal.company.com:8443
Metric Type Description
hawkeye_ssl_expiry_days Gauge Days remaining until the SSL certificate expires. Label: target
hawkeye_ssl_status Gauge 1 = TLS handshake succeeded, 0 = failed. Label: target

8. HTTP Advanced Check

An in-depth HTTP probe that goes beyond simple availability. It measures the full request lifecycle including DNS lookup, TCP connect, TLS handshake, and Time-To-First-Byte (TTFB). Also supports expected HTTP status codes, body pattern matching with regex, and custom headers.

  • Full timing breakdown: DNS, TCP, TLS, TTFB, Total
  • Custom HTTP method, headers, and request body
  • Expected HTTP status code validation
  • Response body regex pattern matching
  • Optional redirect following and SSL verification control
YAMLchecks.yaml — HTTP Advanced example 
http_advanced:
  interval: 60s
  timeout: 15s
  targets:
    - url: https://api.example.com/health
      method: GET
      expectedStatus: 200
      expectedPattern: '"status":"ok"'
      followRedirects: true
      sslVerify: true
    - url: https://auth.example.com/token
      method: POST
      headers:
        Content-Type: application/json
      body: '{"grant_type":"client_credentials"}'
Metric Type Description
hawkeye_http_advanced_status Gauge 1 = check passed (status + pattern matched), 0 = failed. Label: target
hawkeye_http_advanced_latency_seconds Gauge Total end-to-end request latency in seconds. Label: target

The http_advanced check also returns a rich JSON response via GET /v1/metrics/http_advanced containing per-target timing breakdown: dns_ms, tcp_ms, tls_ms, ttfb_ms, total_ms, and status.

Intelligent Alerting

Hawk-Eye integrates with Prometheus Alertmanager for flexible alerting. Alert rules are defined in prometheus_rules.yaml and support all standard Prometheus alert operators.

Supported notification channels include: Email, Slack, PagerDuty, OpsGenie, Microsoft Teams, and any generic webhook.

Dashboards & Visualization

Hawk-Eye ships with pre-built Grafana dashboards for all check types. These are automatically provisioned using the volume mounts defined in docker-compose.yaml.

  • Live Dashboard: Real-time WebSocket-fed status table for all targets
  • Grafana Dashboards: Historical trends, latency heatmaps, SLA tracking
  • Kibana: Deep search across raw logs and OTLP traces

Target Grouping & Filtering

Targets can be organized using Tags or Regex patterns. This makes managing large inventories (50+ targets) much easier by allowing:

  • Filter dashboard by environment: production, staging
  • Group by naming convention using Regex (e.g., ^db-.* to group all database hosts)
  • Active tags persist in browser local storage

Config Export / Import

The full system configuration can be exported as a single ZIP archive, which includes all YAML configs, alert rules, Grafana dashboards, and Alertmanager configs. This enables seamless migration to a new Hawk-Eye instance.

  • Export: Dashboard → Export button → Downloads ZIP
  • Import: Dashboard → Import button → Upload ZIP → Automatic service reload

REST API

All monitoring data is available via a REST API at http://localhost:8080. Key endpoints:

Endpoint Method Description
GET /metrics GET Prometheus-format metrics for all checks
GET /v1/metrics/{check} GET JSON metrics for a specific check (e.g., health, latency)
GET /openapi GET OpenAPI spec (JSON or YAML)
GET /api/v1/targets GET List all configured targets with tags
POST /api/v1/config/export POST Trigger and download a configuration backup ZIP
POST /api/v1/config/import POST Upload and restore a configuration ZIP

All API endpoints (except /login and /metrics) are protected by session cookie authentication.

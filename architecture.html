Product Architecture

An in-depth technical view of the Hawk-Eye system's components, data flows, and integration points.

System Overview

Hawk-Eye is a production monitoring platform developed at T-Systems / Deutsche Telekom. It follows a decoupled, layered architecture where probing, metric storage, and visualization are handled by separate, specialized services. This allows each component to scale independently and be replaced without impacting others.

The system is organized into four architectural layers:

  • Layer 1 — Core Engine: The Go-based Hawk-Eye binary that executes all monitoring checks
  • Layer 2 — Metrics Pipeline: Prometheus for time-series storage and Thanos for long-term, multi-cluster querying
  • Layer 3 — Logging & Tracing: Elasticsearch for log/trace storage, Kibana for visualization
  • Layer 4 — Presentation: Grafana for metric dashboards, Hawk-Eye's own live dashboard for operational view

Architecture Layers

Layer 1: Core Engine (Hawk-Eye)

The central Go binary that orchestrates all probing. It exposes a WebSocket endpoint for real-time dashboard updates, a REST API, and a Prometheus /metrics endpoint. Check definitions are dynamically reloaded from checks.yaml without restart.

Layer 2: Metrics Pipeline

Prometheus scrapes metrics from the Hawk-Eye Engine every 15 seconds and evaluates alerting rules. For long-term metrics retention beyond 15 days, Thanos Sidecar uploads blocks to object storage and Thanos Query federates queries across instances.

Layer 3: Logging & Tracing

Application logs and OpenTelemetry traces are shipped from the Hawk-Eye Engine to Elasticsearch. Kibana provides a UI for full-text search, trace visualization, and incident forensics across the entire log history.

Layer 4: Presentation

Two visualization layers: the built-in Hawk-Eye Dashboard for operators needing a live operational view (no Grafana login required), and Grafana for deep analytical dashboards with historical trends, SLA reports, and annotated events.

Data Flow Walkthrough

Below is a step-by-step trace of a health check probe from execution to user-visible alert:

  1. Hawk-Eye Engine reads checks.yaml at startup and on every reload interval.
  2. A goroutine for each target fires an HTTP GET request at the configured interval.
  3. The response (status code, latency, error) is stored in an in-memory ring buffer and updated as a Prometheus Gauge/Counter.
  4. Prometheus scrapes /metrics every 15s and writes data to its local TSDB.
  5. Prometheus evaluates the alerting rules file every 15s (evaluation interval).
  6. If a rule fires, Alertmanager is notified. It applies routing, grouping, and rate-limiting before sending a notification (e.g., Slack message).
  7. Simultaneously, the Hawk-Eye Engine pushes a WebSocket frame to all connected browser clients, updating the Live Dashboard in real time without a page reload.
  8. OpenTelemetry spans are emitted for each probe and ingested by Elasticsearch, enabling trace-level debugging in Kibana.

Integration Points

Hawk-Eye is designed for extensibility. Key integration interfaces:

  • Prometheus Remote Write: Forward metrics to any Prometheus-compatible backend (Cortex, VictoriaMetrics, Thanos Receive)
  • OpenTelemetry Exporter: Traces can be redirected to Jaeger, Zipkin, or any OTLP-compatible backend via environment variables
  • GitLab Target Discovery: Automatically discover targets from GitLab project metadata using the gitlab loader type
  • Alertmanager Webhooks: Send alerts to any HTTP endpoint for custom automation (e.g., create Jira tickets on alert)
  • REST API: Fully documented API for CI/CD pipeline integration to add/remove targets during deployments

The OpenAPI specification is always available at http://localhost:8080/openapi and can be imported into Postman or Swagger UI for interactive API exploration.

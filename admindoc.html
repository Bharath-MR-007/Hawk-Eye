Admin Document

Technical reference for system administrators responsible for deploying, configuring, and maintaining Hawk-Eye in production environments.

Configuration Files

All configuration files are in the project root directory and are mounted into the Docker container via docker-compose.yaml volumes. Changes to files are read immediately (no restart needed for most files).

File Purpose Hot Reload
checks.yaml Monitoring targets and check intervals ✅ Every 10s
prometheus.yaml Prometheus scrape and global configs ✅ Via API
prometheus_rules.yaml Alerting and recording rules ✅ Via API
alertmanager/alertmanager.yaml Notification channels (Slack, Email, etc.) ✅ Via API
docker-compose.yaml Service definitions and volume mounts ❌ Restart required
grafana/ Grafana dashboard JSON provisioning ❌ Restart required

checks.yaml Full Reference

This file controls all monitoring probes. It is reloaded every 10 seconds — changes are applied to the next check cycle automatically.

YAMLchecks.yaml — full example
# Health Check — verifies HTTP endpoint availability
health:
  interval: 30s       # How often to run the check
  timeout: 10s        # Max time to wait for a response
  retry:
    count: 3          # Retry up to 3 times on failure
    delay: 2s         # Wait 2s between retries
  targets:
    - https://api.production.example.com/health
    - https://auth.production.example.com/ping

# Latency Check — RTT measurement between nodes
latency:
  interval: 60s
  timeout: 10s
  targets:
    - https://hawk-eye-us-east.internal
    - https://api.partner.com

# Standard DNS Resolution Check
dns:
  interval: 60s
  timeout: 5s
  retry:
    count: 2
    delay: 1s
  targets:
    - api.internal.company.com
    - database.internal

# DNS Advanced — multi-resolver, multi-type queries
dns_advanced:
  interval: 60s
  timeout: 5s
  resolvers:
    - 8.8.8.8:53
    - 1.1.1.1:53
  queries:
    - name: api.internal.company.com
      type: A
    - name: mail.company.com
      type: MX

# Traceroute Check — hop-by-hop path analysis
traceroute:
  interval: 300s
  timeout: 3s
  maxHops: 30
  targets:
    - addr: 8.8.8.8
      port: 53
    - addr: gateway.company.com
      port: 443

# TCP Meter — raw TCP handshake timing
tcp_metrics:
  interval: 30s
  timeout: 5s
  targets:
    - database.internal:5432
    - redis.internal:6379
    - api.example.com:443

# SSL/TLS Validity — certificate expiry monitoring
ssl_tls:
  interval: 3600s
  timeout: 10s
  targets:
    - api.example.com:443
    - auth.internal.company.com:8443

# HTTP Advanced — deep probe with timing breakdown
http_advanced:
  interval: 60s
  timeout: 15s
  targets:
    - url: https://api.example.com/health
      method: GET
      expectedStatus: 200
      expectedPattern: '"status":"ok"'
      followRedirects: true
      sslVerify: true
    - url: https://auth.example.com/token
      method: POST
      headers:
        Content-Type: application/json
      body: '{"grant_type":"client_credentials"}'

Configuring Alert Rules

Alert rules are defined in prometheus_rules.yaml. Use the Alerts Config UI in the dashboard to edit them interactively, or edit the file directly.

YAMLprometheus_rules.yaml — example rules
groups:
  - name: hawk-eye-health
    rules:
      # Alert if a target has been DOWN for more than 1 minute
      - alert: TargetDown
        expr: hawkeye_health_up == 0
        for: 1m
        labels:
          severity: critical
        annotations:
          summary: "Target {{ $labels.target }} is DOWN"
          description: "Health check has been failing for over 1 minute."

      # Alert if latency exceeds 300ms
      - alert: HighLatency
        expr: hawkeye_latency_seconds > 0.3
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "High latency to {{ $labels.target }}"
          description: "Latency has been above 300ms for 5 minutes."

      # Alert if SSL certificate expires within 14 days
      - alert: SSLCertExpiringSoon
        expr: hawkeye_ssl_expiry_days < 14
        for: 1h
        labels:
          severity: warning
        annotations:
          summary: "SSL cert for {{ $labels.target }} expires in {{ $value | printf \"%.0f\" }} days"
          description: "Certificate will expire soon. Renew immediately."

      # Alert if SSL check fails (handshake error)
      - alert: SSLCheckFailed
        expr: hawkeye_ssl_status == 0
        for: 5m
        labels:
          severity: critical
        annotations:
          summary: "SSL/TLS handshake failed for {{ $labels.target }}"
          description: "TLS connection could not be established."

      # Alert if TCP port is unreachable
      - alert: TCPPortDown
        expr: hawkeye_tcp_status == 0
        for: 2m
        labels:
          severity: critical
        annotations:
          summary: "TCP port unreachable: {{ $labels.target }}"
          description: "TCP handshake failed — port may be blocked or service is down."

      # Alert if DNS resolution fails
      - alert: DNSResolutionFailed
        expr: hawkeye_dns_status == 0
        for: 2m
        labels:
          severity: warning
        annotations:
          summary: "DNS resolution failed for {{ $labels.target }}"
          description: "DNS check could not resolve the domain."

      # Alert if packet loss exceeds 20%
      - alert: HighPacketLoss
        expr: hawkeye_traceroute_packet_loss_pct > 20
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "High packet loss to {{ $labels.target }} ({{ $value }}%)"
          description: "Network path has significant packet loss detected via traceroute."

Configuring Notification Channels (Alertmanager)

YAMLalertmanager/alertmanager.yaml — Slack example
global:
  resolve_timeout: 5m

route:
  group_by: ['alertname', 'severity']
  group_wait: 10s
  group_interval: 10m
  repeat_interval: 1h
  receiver: 'slack-notifications'

receivers:
  - name: 'slack-notifications'
    slack_configs:
      - api_url: 'https://hooks.slack.com/services/YOUR/WEBHOOK/URL'
        channel: '#hawk-eye-alerts'
        send_resolved: true
        title: '{{ .Status | toUpper }}: {{ .CommonLabels.alertname }}'
        text: '{{ range .Alerts }}{{ .Annotations.description }}{{ end }}'

REST API Reference

The Hawk-Eye API is available on port 8080. All endpoints except /login require a valid session cookie (hawk_session=authenticated).

Method Endpoint Description
GET /metrics Prometheus format metrics (no auth required)
GET /v1/metrics/{check} JSON results for a specific check. Valid values: health, latency, dns, dns_advanced, traceroute, tcp_metrics, ssl_tls, http_advanced
GET /openapi OpenAPI specification (YAML or JSON via Accept header)
GET /api/v1/targets List all targets with tags
POST /api/v1/targets Add a new target
DELETE /api/v1/targets Remove a target
GET /api/v1/alerts List current alert rules
POST /api/v1/alerts Update alert rules
POST /api/v1/config/export Export full config ZIP
POST /api/v1/config/import Import config from ZIP

Environment Variables

Variable Default Description
LOG_LEVEL info Log verbosity: DEBUG, INFO, WARN, ERROR
LOG_FORMAT JSON Log format: JSON or TEXT
HAWKEYE_ENV development Environment label (informational)
HAWKEYE_LOADER_HTTP_TOKEN — Bearer token for remote config URL (http loader)
HAWKEYE_TARGETMANAGER_GITLAB_TOKEN — GitLab API token for auto target discovery

Log Management

Logs are output to stderr in JSON format by default. View them using:

BASH
# Tail live hawkeye logs
docker compose logs -f hawkeye

# Filter for ERROR level only
docker compose logs hawkeye | grep '"level":"ERROR"'

# Save logs to file
docker compose logs hawkeye > hawk-eye-$(date +%Y%m%d).log

Security Hardening

Change the default password (admin/admin) immediately after deployment. The login configuration is in pkg/hawkeye/handlers.go.

  • Run Hawk-Eye behind a reverse proxy (Nginx / Caddy) with TLS in production
  • Restrict access to ports 9090, 9093, 3000, 5601, 9200 to internal VPC only
  • Use HAWKEYE_LOADER_HTTP_TOKEN instead of embedding tokens in YAML files
  • Store exported ZIP backups in encrypted storage (they contain alert webhook URLs)
  • Enable Grafana authentication in grafana/grafana.ini for multi-user environments

Upgrading

  1. Export your current configuration via the Dashboard Export button. Keep this ZIP as a backup.
  2. Pull the latest code: git pull origin main
  3. Rebuild the binary for your target architecture.
  4. Run docker compose up -d --build to recreate the container.
  5. Verify the application is healthy via docker compose ps and curl http://localhost:8080/metrics.

Configuration files (checks.yaml, prometheus_rules.yaml) are persisted via Docker volumes and are not affected by upgrades.
